# 프론트엔드 규칙

1. UI를 만들기 전에 화면 목적을 먼저 정의한다.
2. 디자인보다 정보 구조를 우선한다.
3. "예쁘게", "세련되게", "모던하게" 같은 추상적인 표현을 근거로 구현하지 않는다.
4. 검증된 레퍼런스를 참고하고 새로운 패턴을 임의로 발명하지 않는다.
5. 하나의 컴포넌트는 하나의 책임만 가진다.
6. 500줄 이상의 컴포넌트는 만들지 않는다.
7. 거대한 단일 페이지 파일을 생성하지 않는다.
8. 화면 전체를 수정하지 말고 수정 범위를 명확히 제한한다.
9. 기존 UI를 재설계하지 말고 요청된 부분만 수정한다.
10. 새로운 라이브러리를 임의로 추가하지 않는다.
11. 디자인 토큰(Spacing, Radius, Font)을 임의로 생성하지 않는다.
12. React Native에서는 div, span, button 사용을 피하고 플랫폼에 맞는 컴포넌트를 사용한다.
13. 반응형 대응 없이 데스크톱 전용 UI를 만들지 않는다.
14. 하드코딩된 값보다 재사용 가능한 상수를 우선한다.
15. 사용자의 핵심 행동(CTA)은 항상 명확하게 보이도록 한다.

---

# 바이브 코딩 규칙

1. 코드 작성 전에 설계를 먼저 제안한다.
2. 바로 구현하지 말고 작업 계획을 먼저 설명한다.
3. 추측으로 기능을 추가하지 않는다.
4. 요청되지 않은 기능을 임의로 구현하지 않는다.
5. 기존 코드가 동작하면 이유 없이 리팩토링하지 않는다.
6. 수정 범위를 벗어나는 변경을 하지 않는다.
7. 대규모 리팩토링은 승인 없이 진행하지 않는다.
8. 파일 이름 변경은 반드시 이유를 설명한다.
9. 의존성 추가는 반드시 이유를 설명한다.
10. 모르면 구현하지 말고 질문한다.

# 백엔드 규칙

1. API 설계를 먼저 하고 코드를 작성한다.
2. Request/Response 구조를 먼저 정의한다.
3. 모든 비즈니스 로직을 route 파일에 작성하지 않는다.
4. Route, Controller, Service를 분리한다. Supabase Edge Functions의 경우 index.ts는 라우팅만 담당하고, 비즈니스 로직은 별도 함수/모듈로 분리한다.
5. API 응답 형식은 프로젝트 전체에서 일관되게 유지한다.
6. 에러 처리 로직을 반드시 작성한다.
7. try/catch 없이 외부 API를 호출하지 않는다.
8. API 타임아웃을 고려한다.
9. 외부 서비스 실패 시 대체 응답을 고려한다.
10. DB 접근 코드를 여러 곳에 중복 작성하지 않는다.
11. 로그와 사용자 응답을 구분한다.
12. 데이터 삭제는 Soft Delete 여부를 먼저 검토한다.
13. 비즈니스 로직과 인증 로직을 섞지 않는다.
14. 입력 검증 없이 DB에 저장하지 않는다.
15. 클라이언트 입력값을 그대로 신뢰하지 않는다.

---

# 데이터베이스 규칙

1. 화면 기준이 아니라 데이터 기준으로 설계한다.
2. 테이블 생성 전에 데이터 소유자를 정의한다.
3. user_id 없는 사용자 데이터는 생성하지 않는다.
4. 외래키를 무시하지 않는다.
5. 중복 데이터를 저장하기 전에 정규화를 검토한다.
6. created_at, updated_at을 기본 포함한다.
7. 삭제 정책을 먼저 정의한다.
8. 인덱스 없이 대량 조회를 설계하지 않는다.
9. DB 구조를 프론트엔드 편의성만으로 결정하지 않는다.
10. RLS를 비활성화해서 문제를 해결하지 않는다.

# Supabase 규칙

1. 모든 사용자 데이터는 RLS를 적용한다.
2. service_role key는 서버에서만 사용한다.
3. anon key를 관리자 권한처럼 사용하지 않는다.
4. Storage 파일은 기본적으로 비공개를 고려한다.
5. Signed URL 사용을 우선 검토한다.
6. 사용자가 다른 사용자의 데이터를 조회할 수 없어야 한다.
7. RLS를 우회하는 쿼리를 작성하지 않는다.
8. 관리자 권한은 별도 정책으로 분리한다.
9. Storage 업로드 경로에 사용자 식별자를 포함한다.
10. 정책을 비활성화해서 개발 편의성을 얻지 않는다.

# 보안 규칙

1. API Key는 절대 프론트엔드에 노출하지 않는다.
2. Supabase service_role key는 백엔드에서만 사용한다.
3. 사용자가 보낸 user_id를 그대로 믿지 않는다.
4. user_id는 인증 토큰에서 추출한다.
5. 모든 업로드 파일은 크기 제한을 둔다.
6. 모든 업로드 파일은 MIME Type 검증을 수행한다.
7. 이미지 URL은 public으로 무제한 공개하지 않는다.
8. AI API 호출에는 Rate Limit을 적용한다.
9. 로그인 API에는 추가 Rate Limit을 적용한다.
10. DB Insert/Update/Delete 전 권한 검사를 수행한다.
11. 내부 스택트레이스를 사용자에게 노출하지 않는다.
12. API 에러 원문을 사용자에게 그대로 반환하지 않는다.
13. CORS는 필요한 도메인만 허용한다.
14. 환경변수는 절대 코드에 하드코딩하지 않는다.
15. 비밀번호를 로그에 기록하지 않는다.
16. Access Token을 로그에 기록하지 않는다.
17. Refresh Token을 로그에 기록하지 않는다.
18. SQL 문자열을 직접 조합하지 않는다.
19. 사용자 입력을 그대로 파일명으로 사용하지 않는다.
20. 사용자 입력을 그대로 HTML로 렌더링하지 않는다.
21. 관리자 기능은 일반 API와 분리한다.
22. 인증 없이 비용이 발생하는 API를 호출할 수 없게 한다.
23. 사용량 제한 없이 AI 호출을 허용하지 않는다.
24. 업로드 횟수 제한을 고려한다.
25. 예외 상황에서도 권한 검사를 우회하지 않는다.

4. 테스트되지 않은 코드를 안전하다고 주장하지 않는다.
5. 보안 규칙을 편의상 제거하지 않는다.
6. RLS를 끄는 방식으로 문제를 해결하지 않는다.
7. API Key를 클라이언트에 노출하는 방법을 제안하지 않는다.
8. "임시"라는 이유로 보안 검사를 생략하지 않는다.
9. 추측으로 데이터베이스를 수정하지 않는다.

# VibeScene 프로젝트 규칙

1. 우리 프로젝트의 시스템 프롬프트 v2는 임의로 수정하지 않는다.
2. 플레이리스트 status 흐름은 정해진 순서를 지킨다: pending → analyzing → searching → creating → created (실패 시 failed)
3. Edge Function 1은 'searching' 단계까지만 처리하고, 'creating' 이후는 Edge Function 2에서 처리한다.
4. 디자인 토큰은 src/constants/colors.ts, src/constants/spacing.ts에서만 가져온다.
5. 이미지는 base64로 Edge Function에 전송하지 않고, Storage에 먼저 업로드 후 path만 전달한다.
6. Storage 버킷 이름은 user-images로 통일한다.
7. user_id는 클라이언트가 보내는 값이 아닌, JWT 토큰에서 추출한 값을 사용한다.
8. Supabase 테이블 스키마 변경은 마이그레이션 파일 없이 진행하지 않는다.
9. 곡 검색 실패 시 처리: 5개 미만이면 failed, 5-9개면 rank 재정렬 후 저장.
10. Edge Function 환경변수는 SUPABASE_ 접두사를 쓰지 않는다 (Supabase 예약어).

# 한국어 응답 규칙

1. 사용자와의 모든 커뮤니케이션은 한국어로 한다.
2. 코드는 영어로 작성한다.
3. 주석은 한국어를 우선하되, 영어 표준 용어는 그대로 둔다.
4. 변수명/함수명은 영어로 작성한다.

---

# 작업 흐름 규칙

1. 파일 수정 전에 해당 파일 내용을 먼저 확인한다.
2. 여러 파일을 동시에 수정할 때는 변경 사항을 먼저 요약해서 보고한다.
3. 패키지 설치 시 이유를 설명하고 사용자 승인을 받는다.
4. Git commit 메시지는 한국어로 작성하되, prefix는 영어로 한다 (feat:, fix:, refactor: 등).

# 코드베이스 확인 규칙

1. 구현 전에 관련 파일을 먼저 읽는다.
2. 존재하지 않는 함수, 파일, 테이블을 추측하지 않는다.
3. 현재 구조를 이해하기 전에는 구현하지 않는다.
4. 기존 패턴이 있으면 그 패턴을 따른다.
